Labelle.tn — E-commerce Data Architecture

New entities, their relationships, and how the existing inventory schema is reused
Prepared for the Labelle project  ·  June 12, 2026  ·  Backend: Symfony 5.4 / Doctrine ORM / MySQL  ·  Storefront: Angular (labelle.tn)  ·  Back-office: labelle-inventory-ui

1. The four questions, answered up front

Q1 — Category and sub-category: one entity or two?
ONE ENTITY A single store_category table with a nullable parent_id pointing to itself (an "adjacency list"). Your menu is three levels deep (Visage → Peaux → …) and a self-referencing tree handles any depth with one table, one Doctrine entity, and one set of CRUD screens. Two separate tables would hard-code the depth and break the day the client wants a fourth level or wants to move a sub-category up.
Precedent: PrestaShop (ps_category.id_parent), Sylius (Taxon tree), WooCommerce (term hierarchy), Magento — every major platform models categories as one self-referencing entity. None uses a separate sub-category table.
Q2 — Product details table joined to the existing product table?
YES — ONE-TO-ONE, OPT-IN A store_product_detail table with a unique FK to product. The inventory product table stays the single source of truth for stock and prices; the detail row adds everything the web needs (descriptions, images, slug, categories, publish flag). Crucially, a product is only on the website if a detail row exists and is published — the warehouse can hold 2,410 SKUs while the shop sells a curated 300. Inventory screens and reports are completely unaffected.
Q3 — A separate store_user table for shoppers?
YES — AND LINK IT TO CLIENT Staff and shoppers must never share a credentials table: different auth firewalls, different lifecycle, different risk profile (a leaked shopper password must not be a staff password). One refinement to your idea: keep the existing client table as the customer profile (it already holds every buyer across all channels, keyed by phone), and make store_user the login account that links to a client. A shopper who already bought in the boutique and then registers online gets connected to their full purchase history by phone number.
Precedent: Sylius — the reference Symfony e-commerce framework — splits exactly this way: Customer (profile, may exist without login) vs ShopUser (credentials) vs AdminUser (staff). PrestaShop likewise separates ps_customer from ps_employee. Your client / store_user / user trio maps onto this one-to-one.
Q4 — Web orders: reuse the existing t_order table?
YES — EXTEND, DON'T DUPLICATE Reuse t_order as the single financial and stock record, extended two ways: a new source column (manual | web | intigo) and a one-to-one store_order_detail row carrying web-only data (status, shipping address, fee, payment method). Reasons:
  • Stock: t_order creation already decrements product.quantity atomically. A separate web-order table would need a duplicate of that logic — the #1 source of oversell bugs.
  • Reporting: every KPI, chart, revenue report, and inventory snapshot reads t_order. Web sales appear in the client's existing dashboards on day one, for free.
  • Proven pattern: the Intigo marketplace sync already pushes external orders into t_order. The web shop is just a second external channel.
  • Staff workflow: web orders show up in the existing order list the team uses daily — filterable by source, no second screen to watch.
Precedent: Shopify stores every order from every channel (online store, POS, marketplaces) in one orders table with a source_name column. PrestaShop keeps one ps_orders table and puts state into a separate ps_order_state + ps_order_history pair — which is exactly the status model proposed in §5.

2. Design principles

  • One source of truth. Stock = product.quantity. Sales = t_order. Customers = client. New tables decorate, they never duplicate.
  • Never break the inventory app. Every change to an existing table is additive (new nullable/defaulted columns only). Existing screens, services and reports keep working unmodified.
  • Visual and namespace separation. Every new table is prefixed store_. In the back-office, the e-commerce module lives in its own sidebar group; in the schema, the prefix gives the same separation.
  • Snapshot what history must not rewrite. Orders copy the shipping address, prices, fees and discount amounts at purchase time. Editing a customer's address must never change a shipped order. (t_order already does this for client name/phone — the pattern is extended, not invented.)
  • Proper utf8mb4 from day one. All new tables are created with utf8mb4_unicode_ci so the legacy Utf8Encoder workaround is never needed on the new API surface.
  • Money columns are DECIMAL(10,3). The Tunisian dinar has 3 decimals (millimes). Legacy float columns stay as they are; new ones do it right.
  • Deliberately no EAV, no variants engine. Magento's attribute-value model is the canonical example of flexibility nobody asked for becoming everyone's performance problem. Each sellable item here is one inventory product with its own barcode — which is how the warehouse already works. See §8 for the escape hatch if variants are ever needed.

5.3 store_order_status + store_order_status_history

Statuses live in a table, not an enum, with behavior flags — so adding a state ("awaiting exchange") or changing which state triggers an email is data entry, not a deployment.

store_order_statusTypeNotes
idINT PK AI
codeVARCHAR(32) UNIQUEStable machine name used in code and as frontend translation key.

Precedent: this is PrestaShop's ps_order_state almost verbatim — its rows carry paid, shipped, delivery, send_email flags that drive behavior, and ps_order_history records every transition. It is the most battle-tested order-state model in e-commerce.

Seed statuses: pending (new, stock already taken) → confirmed (staff called the customer — the COD norm in Tunisia) → shipped (sets_delivered? no — tracking set) → delivered (sets_delivered + sets_payment_received for COD) → terminal; plus cancelled and returned (both restocks=1).

5.4 Checkout flow — how a web order is born

  • Checkout calls the existing order-creation service (the same code path used by manual orders and Intigo), with source='web'. Stock decrements, sold_units updates, the client is upserted by phone — all existing, tested logic.
  • The same transaction inserts store_order_detail (status pending, address snapshot, shipping fee from the zone table) and the first history row.
  • Stock is taken at order creation — identical to today's behavior, so two simultaneous shoppers cannot buy the last unit. Cancellation restocks via the restocks flag.
  • The existing 5-second duplicate-order guard protects against double-submit on the web too.
  • Order confirmation email is queued (§6.4).

6.3 store_cart / store_cart_item + store_wishlist_item

9. Deliberate simplifications (and their escape hatches)

Not buildingWhyIf it's ever needed
Product variants engine (size/shade matrices)Each variant already exists as its own inventory product with its own barcode — that's how the warehouse scans and counts. A variants layer would duplicate stock-keeping.Add variant_group VARCHAR NULL on store_product_detail; storefront groups rows sharing it into one page with a selector. No migration of existing data needed.
EAV / dynamic attributes (Magento-style)Killed many shops' performance for flexibility a 300-product beauty store never uses.A nullable JSON attributes column on store_product_detail covers "Contenance: 50ml" display tables.
Multi-currency / multi-language contentTND only; content in French (UI chrome is already translatable in the Angular apps).Add store_product_detail_translation later; slugs already unique per row.
Reserved-stock / inventory holdsStock decrements at order creation (existing behavior), so nothing needs reserving.Only relevant if checkout ever stops creating orders immediately (e.g. pay-first flows) — then add a reserved_until on cart items.
Server-side guest cartslocalStorage works today, zero backend load, no session GC.store_cart.store_user_id is already nullable-ready; add a token column and a cookie.

10. Decisions to confirm before Phase 1 starts

  • Image storage: proposed = backend disk under public/uploads/products/ (same pattern as Documents), resized at upload. Confirm hosting disk space is adequate for ~300 products × 2–3 images.
  • Brand entity in Phase 1? Cheap now; confirm the client wants brand filtering/pages.
  • Who curates content? Publishing 300 products (photos, descriptions) is the real project bottleneck — decide who does data entry and whether to import anything from the old PrestaShop database (fpmiqfdv_inventory.sql sits in the workspace; the old shop's ps_product_lang descriptions and images may be importable, saving weeks).
  • Order numbering: web orders share t_order.id sequence with manual orders. If the client wants "WEB-1024"-style references, add a display-reference column on store_order_detail — decide before first real order.
  • Prod PHP upgrade (7.3 → 8.1/8.2) before the public API goes live — already discussed; affects dependency security patching, not this schema.
